# Horror Courier Simulator - Development Guidelines

## Project Info
- **Engine**: Unity 6 (6000.0.71f1) with HDRP
- **Input**: New Input System (1.19.0) via InputManager singleton
- **Language**: C# 9, .NET Standard 2.1

## Architecture Rules

### Input
- All input goes through `InputManager.Instance.actions`
- Never use `Input.GetKey()` / `Input.GetAxis()` (old system)
- Read input via `InputAction.ReadValue<T>()` or `InputAction.IsPressed()` in Update
- Do NOT subscribe lambdas to input events (they can't be unsubscribed)

### Components
- One responsibility per MonoBehaviour
- Use `[RequireComponent]` when a script depends on another component
- Use `[SerializeField] private` for inspector fields, not `public`
- Expose read-only state via properties: `public float Value { get; private set; }`

### Communication Between Systems
- Use C# `event Action<T>` or UnityEvent for loose coupling
- Avoid direct references between gameplay systems
- Singletons only for truly global services (InputManager, GameManager, AudioManager)

### Interaction
- Interactable objects implement `IInteractable` interface
- Player detects them via Physics.Raycast from camera

### Data
- Game data (items, routes, horror events) lives in ScriptableObjects
- ScriptableObjects go in `Assets/Script/Data/`

## Code Style
- PascalCase for public members, methods, properties, classes
- camelCase for private fields (with no underscore prefix)
- `[Header("Section")]` to group serialized fields in inspector
- Keep MonoBehaviours small — extract logic into plain C# classes when it grows

## Folder Structure
```
Assets/Script/
├── Core/           # Singletons, managers
├── Player/         # Movement, camera, interaction, inventory
├── Systems/        # Delivery, horror, dialogue
├── AI/             # NPC logic
├── UI/             # HUD, menus
├── Interactables/  # IInteractable implementations
└── Data/           # ScriptableObjects
```

## Common Mistakes to Avoid
- Do NOT use `new` to create MonoBehaviours — use AddComponent or Instantiate
- Do NOT read input in FixedUpdate (use Update, cache for FixedUpdate if needed)
- Do NOT forget to disable InputActions in OnDisable
- Do NOT put camera logic in Update — use LateUpdate to avoid jitter
- The auto-generated `InputSystem_Actions.cs` is regenerated by Unity — never edit it manually
